## 第一部分：项目的演进之路 (Phase 1-4)

本项目并不是一蹴而就的，而是经历了一个从“基础对话”到“高级智能体”的四个硬核演进阶段。

### 📌 阶段一：让 AI 拥有“手和脚” —— Agent 工具调用 (Function Calling)

**1. 为什么要做这个功能？**
大模型（LLM）的知识是停留在其被训练的那个时间的（比如截止到 2023 年），并且它无法感知外部世界。如果你问它“今天北京天气如何？”或“现在几点了？”，它要么胡编乱造，要么告诉你它不知道。
为了让 AI 能解决实际问题，我们需要给它装上“手和脚”，这就是 **Agent（智能体）** 的核心概念。

**2. 核心技术点与实现：**
- **Function Calling（函数调用）**：这是 OpenAI 提出的一种标准协议。我们在代码中定义了工具接口 `ToolHandler`，并实现了 `WeatherTool`（查天气）和 `TimeTool`（查时间）。
- **流程解析（保姆级拆解）**：
  在传统的 HTTP 请求中，一问一答就结束了。但在流式（SSE）与工具调用结合的场景下，其交互链路非常复杂。我们把这个过程想象成**“老板（前端）安排秘书（后端）去问专家（大模型）问题”**，分为以下 6 个步骤：
  
  **第一步：项目启动时的“自动登记”（Spring 核心原理）**
  *💡 为什么项目启动时就会扫描工具？*
  这是因为我们利用了 Spring Boot 的 **IoC（控制反转）和依赖注入（DI）** 机制。
  1. 我们在 `WeatherTool` 和 `TimeTool` 类上面加了 `@Component` 注解。这个注解的意思是告诉 Spring：“嗨，我是一个组件，项目启动时请把我实例化并放到你的容器（Bean 容器）里。”
  2. 我们写了一个 `ToolRegistry`（工具注册中心）类，它的构造函数里写了 `List<ToolHandler> handlers`。
  3. 项目启动时，Spring 看到 `ToolRegistry` 需要一堆 `ToolHandler`，它就会自动去容器里把所有打过 `@Component` 标签的工具（天气、时间）全部收集起来，打包塞给 `ToolRegistry`。
  *结果*：就这样，后端在还没开始聊天前，就已经把所有可用工具整理成了一本“工具说明书”（JSON 格式）。

  **第二步：专家决定求助工具（模型决策与流式返回）**
  当用户（老板）提问：“现在几点了？”
  后端（秘书）把问题和那本“工具说明书”一起发给大模型（专家）。
  专家一看：“我脑子里没有当前时间啊，但我看到说明书里有个 `get_current_time` 工具，我得用它！”
  此时，专家**不会直接给出答案**，而是通过 SSE（一种像流水一样一点点传数据的技术）传回一段特殊的指令：`tool_calls`。
  *大坑预警*：因为是像流水一样传，参数（`arguments`）是被切碎成无数个小片段（Delta）发过来的，比如：
  第一秒发来：`{"arguments": "{\""}`
  第二秒发来：`{"arguments": "city\""}`

  **第三步：秘书悄悄拼凑碎片（拦截与拼接）**
  后端（秘书）在接收流水时，发现专家不是在回答问题，而是在要工具！
  此时，后端会**扣下这些碎片，不把它们发给前端**（不然前端界面上就会出现乱码一样的 JSON 字符串）。
  后端准备了一个“胶水”（Java 里的 `StringBuilder`），耐心等待，把专家发来的每一个碎片都拼接到一起，直到专家说“我发完了”。此时，后端手里就拿到了一段完整的 JSON 参数。

  **第四步：秘书自己动手跑腿（本地执行与异常降级）**
  拿到完整的工具名和参数后，后端（秘书）开始执行本地的 Java 代码（比如去调取系统的 `LocalDateTime.now()`）。
  *高频面试考点（容错机制）*：如果专家（大模型）抽风了，发过来的 JSON 少了一个括号，导致解析报错怎么办？
  我们在代码里加了 `try-catch`（异常捕获）保护伞。如果解析失败，后端不会崩溃，而是返回一个“解析错误”的提示，强行继续走下一步。这就叫**优雅降级**。

  **第五步：把结果塞给专家，再问一次（组装历史与递归重发）**
  后端拿到了时间（比如“19:00”），现在必须要把结果告诉专家，让他总结。
  于是，后端在内存里悄悄伪造了两条聊天记录：
  1. 专家刚才的请求记录（证明专家确实要了这个工具）。
  2. 角色为 `tool` 的消息，内容是：“系统时间是19:00”。
  然后，后端**在代码内部再次调用自己（这就叫递归）**，带着这个结果重新向大模型发请求。注意，这一切都是在极短的时间内发生的，前端根本不知道后端和专家私下里又聊了一轮。

  **第六步：拿到最终答案（最终答复与递归熔断）**
  专家收到带时间的聊天记录后，恍然大悟，开始正常生成文本：“现在是晚上 19:00。” 此时，后端才把这些正常的文字像流水一样推给前端。
  *高频面试考点（防死循环）*：万一专家是个“傻子”，拿到结果后又要求调工具，一直无限循环怎么办？
  我们在代码（`streamChatWithRecursionLimit`）里加了一个“保险丝”（递归深度限制）。如果专家连续要工具超过 5 次，后端就会强行切断（熔断），报错停止。这就防止了服务器内存被耗尽（OOM）。

### 📌 阶段二：让 AI 拥有“企业大脑” —— RAG 深度调优

**1. 为什么要做这个功能？（什么是 RAG？）**
大模型就像是一个读了很多书但**没有联网、也没去过你们公司**的学霸。如果你问它“咱们公司今年的年假怎么休？”，它肯定不知道，甚至会胡编乱造（这就叫**AI 幻觉**）。
RAG（检索增强生成）就是给这个学霸配一个**开卷考试的图书馆**。当用户提问时，我们先去图书馆（向量数据库）里把相关的规定找出来，跟问题一起递给学霸，学霸照着资料回答，这样就不会出错了。

**2. 核心技术点与流程解析（保姆级拆解）：**

  **第一步：把厚书切成薄纸片（文档解析与科学分块 Chunking）**
  *💡 为什么不把一整本 PDF 直接丢给大模型？*
  因为大模型一次能看的字数（上下文窗口）是有限的，全塞进去不仅贵得离谱，它还会“看前面忘后面”。
  - 我们用了一个叫 `Apache Tika` 的工具，它像一个扫描仪，把 PDF、Word 里的文字全扣出来。
  - 然后用 `TokenTextSplitter` 像切西瓜一样把几万字的文章切成一小块一小块（比如规定每块 800 个字）。切的时候还特意留了点“重叠部分”，防止一句话刚好被从中间切断，导致前后文连不上。

  **第二步：给纸片打上条形码存起来（向量化与存储）**
  切好的文字纸片计算机是看不懂的，我们要把这些文字变成一串串超长的数字（比如 `[0.1, -0.5, 0.8...]`），这个过程叫**向量化 (Embedding)**。
  这些数字（向量）被存进了 Redis 数据库里（项目里用了 Spring AI 的 `RedisVectorStore`）。

  **第三步：双管齐下找资料（多路召回 Hybrid Search）**
  当用户提问：“苹果手机怎么用？”时，系统怎么去数据库里找相关的纸片呢？我们用了两招：
  - **招式A（向量语义检索）**：把用户的问题也变成一串数字去库里比对“距离”。距离越近，意思越像。它很聪明，搜“苹果”，它能联想到“iPhone”、“乔布斯”。
  - **招式B（BM25 字面检索）**：招式 A 虽然聪明，但有时候遇到专有名词（比如“产品代号 XT-908”）容易瞎联想。所以我们又加了最传统的关键字匹配，文章里有这几个字才被搜出来。
  *结果*：系统同时派这两路人马去搜，各自找回几张纸片，然后把它们合并在一起（去重），这叫**多路召回**。

  **第四步：优中选优（交叉重排 Rerank）**
  刚才找回来的纸片可能有 10 张，但最精华的其实只有 3 张。怎么挑？
  我们请了一位专门做阅读理解的老师——**智谱的 `rerank-1` (BGE-Reranker) 模型**。把这 10 张纸片和用户的问题丢给它，它会仔细对比，给每张纸片打个精确的分数，然后只留下得分最高的 Top 3。
  *高频面试考点（高可用容错）*：万一这位阅读理解老师（重排 API）今天请假不上班（网络报错 400/500）怎么办？
  我们在代码里加了 `try-catch`。如果重排报错，我们不报错，而是**直接把刚才找回来的 10 张纸片里的前 3 张拿来凑合用**。这就保证了系统即使遇到外网故障，聊天功能也绝对不会瘫痪。

### 📌 阶段三：控制成本的艺术 —— Token 精算与滑动窗口

**1. 为什么要做这个功能？**
调用商业大模型（比如 GPT-4, GLM-4）是**按字数（Token）收钱**的。
如果用户聊了 100 轮，第 101 次提问时，如果把前面 100 轮聊天记录全发给 AI，那这一次请求可能就要花掉好几块钱！而且大模型的记忆容量有限，一旦超出限制就会直接报错。

**2. 核心技术点与流程解析（保姆级拆解）：**

  **第一步：算盘打得精（精准 Token 计算）**
  *💡 Token 是什么？*
  Token 不是简单的“一个汉字”或“一个英文字母”。大模型有一套自己的词典，比如“hamburger”可能算 1 个 Token，而一个生僻汉字可能算 3 个 Token。
  如果我们在 Java 里用 `text.length()` 去估算，误差会极大。
  所以我们引入了一个叫 `jtokkit` 的第三方包。它内置了大模型同款的字典算法（CL100K_BASE 编码），能在我们发请求前，**在 Java 本地一分不差地算出这句话到底要消耗多少 Token**。

  **第二步：丢车保帅（滑动窗口截断 Sliding Window）**
  以前的做法很傻：固定只发最近的 8 条聊天记录，之前的直接扔掉。
  现在我们升级成了“精细化截断”：
  - 假设我们规定每次请求最多只能发 7000 个 Token 的内容。
  - 发送前，系统先把当前所有聊天记录算一遍总数。如果不超 7000，直接发。
  - 如果超了 7000，系统会进入**“滑动窗口”模式**：它会把聊天记录按重要程度分类。
  - **最重要不能丢的**：系统的提示词（比如“你是一个友好的助手”），以及工具调用的结果（不然 AI 会犯傻）。
  - **可以牺牲的**：很久以前的普通闲聊记录。
  系统会从最老的闲聊记录开始，一条一条地删，边删边算 Token，直到总数降到 7000 以下为止。这就保证了在不报错、不超预算的前提下，尽可能多地给 AI 提供上下文。

### 📌 阶段四：上帝视角 —— 运营管理大盘与权限控制

**1. 为什么要做这个功能？**
一个做给企业用的系统，不能只有前台给普通员工聊天用。老板和管理员需要一个“上帝视角”，能看到谁在用、用了多少钱、系统健不健康。

**2. 核心技术点与流程解析（保姆级拆解）：**

  **第一步：发身份证（RBAC 权限体系与 JWT）**
  *💡 怎么区分普通员工和老板？*
  我们在数据库的 `user` 表里加了一个 `role`（角色）字段。普通人是 `user`，老板是 `admin`。
  当用户登录时，后端不仅校验密码，还会生成一张带有防伪签名的数字身份证——**JWT (JSON Web Token)**。这张身份证的肚子里（Payload）就偷偷写着这个人的角色。
  
  **第二步：门口的保安（JwtInterceptor 拦截器）**
  当有人想调用看大盘的接口（`/admin/stats`）时，请求会先经过 Spring Boot 的拦截器（就像大楼的保安）。
  保安会检查请求头里有没有 JWT 身份证，并且会用服务器的私钥去验真假。如果是真的，且里面写着 `admin`，保安才会放行；如果是 `user` 或者伪造的身份证，保安直接打回报错（401 无权限）。

  **第三步：算总账（数据大盘与 Token 消耗追踪）**
  得益于我们在阶段三做的 `Token 精算`，现在用户每次聊完天，系统都会在后台把这次消耗的 Token 累加到 `user` 表的 `total_tokens` 字段里。
  管理员调接口时，后端只要去数据库里查一查，就能排出“全公司最爱和 AI 聊天（最烧钱）的员工排行榜”，把看不见的 AI 调用转化为了可视化的运营成本数据。

## 第二部分：核心技术概念扫盲

在面试或与他人交流本项目时，以下这些高频词汇你必须能倒背如流：

### 1. SSE (Server-Sent Events)
- **是什么**：一种允许服务器单向向客户端推送数据的 HTTP 技术。
- **项目中用在哪**：AI 打字机效果。AI 蹦出一个字，后端就用 SSE 推给前端一个字，大大降低了用户等待时间（首字响应时间 TTFT）。
- **对比 WebSocket**：WebSocket 是双向的（前端后端都能互相发），比较重；SSE 是单向的（仅后端推给前端），基于纯 HTTP 协议，非常适合做大模型流式输出。

### 2. Reactor / WebFlux (响应式编程)
- **是什么**：Spring 体系下的一种非阻塞、异步的编程模型。
- **项目中用在哪**：处理 AI 的流式数据。传统的 Spring MVC 是“一个请求一个线程”，如果 AI 生成需要 10 秒，这个线程就卡死 10 秒。WebFlux 采用事件驱动，用极少的线程就能扛住几万人的并发流式对话。代码里随处可见的 `Flux<String>` 就是它的标志。

### 3. Redis Vector Store 与 BM25
- **向量检索 (Vector Search)**：把文本变成一串高维数字（如 [0.1, -0.5, 0.8...]），然后计算两个数组的距离（余弦相似度）。距离越近，代表它们在“语义”上越相似。比如搜索“苹果”，它能搜出“乔布斯”。
- **BM25 检索**：基于词频统计的经典搜索算法。搜索“苹果”，它就只找文章里出现“苹果”这两个字的段落。
- **为什么两者结合叫 Hybrid Search（混合检索）**？因为向量搜索虽然懂语义，但遇到产品型号、专有名词容易跑偏；BM25 找专有名词极准。两者互补，天下无敌。

### 4. JWT (JSON Web Token)
- **是什么**：一种无状态的身份认证机制。
- **项目中用在哪**：
  - 用户登录系统时颁发身份凭证（包含 userId 和 role）。
  - 在调用智谱 AI 的 Rerank 接口时，我们需要自己用 HS256 算法签名生成 JWT 发给智谱服务器，证明“我们是合法的调用者”。

## 第三部分：项目文件结构导读

了解以下核心类，你就能掌控整个项目的脉络：

- `com.niit.agent.controller.ChatController`：核心对话入口，处理 SSE 连接、心跳保活、异常下发。
- `com.niit.agent.service.model.AbstractOpenAiCompatibleModel`：最硬核的底层类。负责管理 OkHttp 连接池、解析大模型返回的碎片化 JSON、拼接 Tool Calls 并在递归执行后返回。
- `com.niit.agent.service.impl.MemoryServiceImpl`：RAG 检索引擎的大脑。负责双路召回、调用重排服务、构建并组装发送给 AI 的终极上下文。
- `com.niit.agent.service.impl.TokenServiceImpl`：精算师。负责调用 JTokkit 精准计算消耗，并执行滑动窗口切断。
- `com.niit.agent.service.tools.ToolRegistry`：百宝箱。自动扫描所有标注了 `@Component` 并实现 `ToolHandler` 接口的类，注册为大模型可用的工具。
